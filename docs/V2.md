# $SCRATCH v2 — generation plan

Planning doc for the next contract generation. Not a buildspec; decisions below are binding unless explicitly reopened.

## Decided

### StakingVault v2 — unbonding instead of burn-on-withdraw

- Replace immediate withdraw (which burns pending + banked tickets) with **request → cooldown → claim**.
- **Tickets survive** through the unbond window (the burn-on-any-withdrawal invariant is intentionally retired for v2).
- Unbond **duration TBD** — candidate range **24–48h**.

### ScratchGame v2 — `scratchMany`

- Add `scratchMany(tier, count)` for **one-signature multi-scratch** in every wallet (site already pipelines sequential scratches when the wallet cannot batch).
- Same tier per call; tickets burned / requests issued atomically under the existing CEI + reentrancy posture.

### Wiring: freeze-on-first-set vs narrow rewire valve

**Freeze-on-first-set (current posture):** once `setGame` / ticket-source wiring is set, it cannot be changed. Maximizes “no admin path to redirect deposits/tickets” optics; mistakes require a parallel redeploy.

**Narrowly-scoped owner rewire valve:** allow owner (treasury) to retarget a specific wiring seam (e.g. game ↔ vault) under the existing 48h timelock pattern, with loud events. Faster incident response; slightly weaker “immutable wiring” story.

v2 should pick one explicitly in the buildspec before coding.

## Migration plan (from wiring report)

Full **parallel stack** redeploy — do not in-place upgrade:

1. Deploy new: ScratchGame + PrizeVault + StakingVault + StandardTicketSource + randomness provider (Pyth adapter or fresh SelfEntropy).
2. **Prize inventory:** announce → queue 48h PrizeVault `sweep` timelock → **3-day spine** (public notice window) → execute sweep + fund new vault + flip site/config to new game.
3. **Stakers:** must **re-stake** into StakingVault v2 — warn **scratch-tickets-first** (spend or accept loss of old-vault ticket economics before migrating principal).
4. **Standard tickets:** re-grant from a holder/eligibility **snapshot** after the new source is live.
5. **Drain all pendings** on the old game (reveal or rescue) **before** cutover; do not orphan mid-flight requests without a published rescue plan.
6. **Fresh entropy chain** if still on SelfEntropy; register commitment before enabling scratches.
7. Checklist: **provider ownership** (and callback) on the new stack; treasury Ownable2Step transfers; site + dashboard addresses; operator `CHAIN_FILE` / Render cutover.

## Not changing

- Token (`$SCRATCH`)
- Tier model (standard / premium separation, non-fungible ticket sources)
- Odds model (1_000_000 cumulative, terminal no-win)
- Transparency posture (events, public tables, ledger / settle visibility)
